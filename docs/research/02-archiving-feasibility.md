# 콘텐츠 '자체 저장'의 기술적 현실성 조사 (2026-07 기준)

## 0. 총론 — 조사 전체를 관통하는 결론

"서버가 URL을 받아 대신 콘텐츠를 가져오는" 클래식 아키텍처는 X·LinkedIn·YouTube 세 플랫폼 모두에서 2024~2026년 사이에 사실상 붕괴했다. 반면 일반 웹페이지와 음악 메타데이터는 성숙한 무료/오픈소스 경로가 건재하다. 따라서 아카식 레코드의 캡처 파이프라인은 소스별로 이원화가 불가피하다: **로그인 장벽 플랫폼은 사용자의 브라우저(확장/공유시트)에서 캡처 → 서버는 저장·AI 분류만 담당**, 일반 웹/음악은 서버 사이드 인제스트 가능.

## 1. X(트위터) 게시물 저장

### 공식 API 현황 (검증됨, 단 3rd-party 블로그 기반 — 아래 주의)
- 2026년 2월 6일부터 X는 티어 구독제를 폐지하고 **pay-per-use가 기본**: 포스트 읽기 $0.005/건, 쓰기 $0.015/건(링크 포함 포스트는 $0.20), 월 읽기 상한 200만 건. **Free tier는 신규 개발자에게 완전 폐지**.
- Basic($200/월)·Pro($5,000/월)는 신규 가입 불가, 기존 구독자는 2026년 6월 1일부터 pay-per-use로 강제 이전 중. Enterprise는 약 $42,000/월부터.
- 시사점: 사용자당 월 수십 건 저장이라도 read당 $0.005는 "저장 시점 1회 read"로는 감당 가능($0.005/포스트)하나, OAuth 앱 심사·계정 관리 부담과 정책 변동성이 크다. 무료 티어가 없어 프리미엄 전환 전 무료 사용자에게 원가가 발생한다.
- 주의: 위 수치는 postproxy.dev, xpoz.ai, twitterapi.io 등 복수의 독립 소스가 일치했으나 X 공식 pricing 페이지 원문 대조는 못 했다(공식 docs.x.com 가격 페이지 직접 확인 필요).

### 비로그인 스크래핑
- **Nitter 생태계는 사망**: 2024년 1월 X가 게스트 계정 생성을 차단하며 공식 인스턴스 폐쇄. 2025년 2월 개발 재개 발표가 있었지만 실계정 풀(account pool)을 넣어 돌리는 셀프호스트 형태만 동작 — 계정 정지 리스크를 상시 안고 가는 구조.
- **twscrape / twikit / Scweet**: 실계정 쿠키+내부 GraphQL API 기반으로 여전히 동작. 다만 계정 풀 확보·정지 대응이 운영 비용이고, ToS 위반이라 상용 서비스의 코어 경로로는 부적합.

### 실제로 동작하는 무인증 우회 경로 2개
1. **공식 oEmbed** (`publish.twitter.com/oembed`, publish.x.com): 인증 불필요, x.com URL 지원, 현재 동작. 단일 포스트의 텍스트·작성자·HTML을 JSON으로 반환. 2026년 현재도 살아있으나 간헐적 실패가 보고되고, 렌더링용 widgets.js 의존이라 "원문 텍스트 보존" 용도로는 blockquote 안의 텍스트를 파싱해 쓰는 형태가 된다.
2. **Syndication API** (`cdn.syndication.twimg.com/tweet-result`, Vercel react-tweet가 사용): 텍스트·이미지·비디오 URL·인용 트윗까지 구조화된 JSON. 2025년 8월 기준 동작 확인 보고가 있으나 **비공식 — 랜덤 빈 응답(200에 빈 바디), 프로덕션 IP 차단 사례 보고됨**. 파생으로 **FxTwitter/FxEmbed API**(오픈소스, MIT, `api.fxtwitter.com`)가 미디어 직링크까지 주는 공개 API를 제공하며 셀프호스트도 가능.

### 기존 툴들의 실제 방식
- **Readwise**: X 공식 API 연동(멘션 `@readwiseio save`, DM, 좋아요 트리거) — 즉 API 비용을 감수하는 모델.
- **dewey., Twillot, BookmarkSave**: 전부 **브라우저 확장** — 사용자가 로그인한 세션에서 북마크 페이지의 DOM/내부 XHR 응답을 읽어 추출. X 입장에서 차단할 방법이 사실상 없고(사용자 본인 트래픽), 미디어 파일 다운로드까지 지원. 2026년 현재 이 카테고리가 가장 활발.

### 권장 접근
- **1순위: 브라우저 확장 캡처** — 사용자가 보고 있는 포스트의 DOM에서 텍스트·작성자·미디어 URL을 추출해 서버로 전송. 스레드 전체 캡처도 확장에서만 안정적으로 가능. 법적으로도 "사용자 본인의 브라우저에서 본인이 보는 콘텐츠 저장"이라 서버 사이드 스크래핑과 리스크 등급이 다르다.
- **2순위(폴백, URL만 공유된 경우): oEmbed → FxTwitter API 순서로 서버 사이드 시도.** 미디어 원본은 캡처 시점에 즉시 다운로드해 자체 스토리지에 복제(트윗 삭제·계정 폐쇄 대비).
- 공식 API는 스킵. 비용 자체보다 무료 티어 부재 + 정책 변동성이 문제.

## 2. LinkedIn 게시물 저장

### API
- 공식 API(Community Management/Posts API)는 **승인제 파트너 전용**. Marketing Developer Platform 심사는 수 주~수 개월 걸리고 다수 거절. 승인받아도 "social feed use case"에 데이터 사용 금지 조항이 있어 아카이브 서비스는 사실상 승인 대상이 아니다. 2025년 Member Post Analytics API가 열렸지만 이는 본인 게시물 성과 지표용.

### 법적 지형 (중요)
- **hiQ v. LinkedIn의 최종 결말은 hiQ 패소**: "공개 프로필 스크래핑은 CFAA 위반 아님"(9th Cir. 2022)이라는 유명한 판시에도 불구하고, 2022년 11월 지방법원이 유저 계약(ToS) 위반을 인정, 12월 $500,000 배상 + 영구 스크래핑 금지 + 데이터·소스코드 전량 파기로 종결. 즉 "CFAA는 피해도 계약 위반으로 죽는다"가 확립된 선례.
- **Proxycurl 사망(2025)**: LinkedIn이 2025년 1월 제소(계약 위반·CFAA·사기 등 6개 청구), 2025년 7월 4일 영구 서비스 종료 + 영구 금지 명령. 부트스트랩 기업이 소송 비용을 못 버티고 접었다고 창업자가 공개 회고. LinkedIn이 스크래핑 업체를 상대로 본보기 소송을 실행하는 플랫폼임이 재확인됨.
- 결론: **서버 사이드 LinkedIn 스크래핑(가짜 계정·헤드리스 브라우저)은 상용 서비스로서 감수 불가능한 리스크.**

### 현실적 캡처 방법
1. **브라우저 확장** — X와 동일한 논리. 사용자가 로그인한 본인 세션에서 보는 게시물의 DOM 캡처. LinkedIn의 anti-bot이 강해도 사용자 본인 브라우저 내 동작은 차단 대상이 아니며, hiQ/Proxycurl형 리스크(대규모 자동 수집)와 법적 성격이 다르다. (단, LinkedIn ToS는 확장 프로그램에 의한 데이터 추출도 금지 문구가 있으므로 "사용자 개인 보관용" 포지셔닝과 볼륨 제한이 방어 논리가 된다 — 이 부분은 법률 자문 필요.)
2. **공식 embed iframe** — 공개(Public) 게시물은 누구나 "Embed this post"로 iframe 코드를 얻을 수 있고 무료·무제한. 단 이것은 '링크'이지 '보존'이 아님(원본 삭제 시 iframe도 죽음). 렌더링 보조용으로만 가치.
3. 미확인: LinkedIn은 oEmbed 공개 엔드포인트가 없다(검색에서 발견 못 함).

### 권장 접근
- **브라우저 확장 캡처 단일 경로로 설계.** 모바일은 공유시트로 URL만 받고 "다음에 데스크톱에서 열면 본문을 캡처한다" 같은 지연 캡처 UX가 현실적. 서버 사이드 수집은 어떤 형태든 배제. 공개 게시물은 embed iframe을 '라이브 뷰' 보조로 병용.

## 3. 유튜브

### yt-dlp 현황
- yt-dlp 자체는 활발히 유지되나, YouTube가 **PO Token(Proof of Origin)** 체계를 강제 — 없으면 403/봇 체크. 토큰 수명은 짧고(12시간 수준 보고), bgutil-ytdlp-pot-provider 같은 외부 프로바이더 플러그인 생태계로 대응 중. "토큰이 있어도 차단 미보장"이 공식 위키의 표현.
- **데이터센터 IP는 광범위 차단**: AWS/GCP/Azure/DO IP 대역에서 "Sign in to confirm you're not a bot" 빈발. 서버에서 돌리려면 residential proxy 회전 + 계정 쿠키가 필요 — 비용·계정 정지 리스크·불안정성 모두 높음.

### 법적/약관 리스크
- YouTube ToS는 다운로드 명시 금지. **Yout v. RIAA**에서 지방법원은 스트림 리핑 서비스가 DMCA §1201 기술적 보호조치 우회에 해당할 수 있다고 판단(Yout 패소, 항소 진행 중, 2025년 10월에도 항소심 브리핑 계속). 영상 파일을 서버에서 대신 받아 보관해주는 상용 서비스는 DMCA 표적이 되는 카테고리다.
- 공식 YouTube Data API를 쓰면 반대 방향의 제약: **비인증 데이터 30일 초과 저장 금지**(삭제 또는 리프레시 의무) — "영구 아카이브" 컨셉과 정면 충돌. 또 captions.download는 본인 소유 영상만 가능.

### 자막 확보
- youtube-transcript-api(비공식)로 자동생성 자막 추출 가능하나 **클라우드 IP에서 RequestBlocked 일상화** — residential proxy(Webshare 등) 통합이 사실상 표준 우회법. 이 역시 서버 사이드는 불안정.
- 브라우저 확장이라면 사용자 세션에서 자막 트랙(timedtext) 요청을 그대로 읽을 수 있어 차단 문제가 없다.

### 권장 접근
- **영상 원본 저장은 기본 기능에서 제외** (기술적 불안정 + DMCA 리스크 + 스토리지 비용 삼중고).
- 저장 대상: **메타데이터(제목·채널·설명·게시일) + 썸네일 이미지 + 자막/트랜스크립트**. 이것만으로 AI 분류·검색·"링크가 죽어도 내용 파악"이라는 핵심 가치의 90%가 나온다.
- 확보 경로: 메타데이터는 YouTube oEmbed(무인증)나 Data API, 자막은 브라우저 확장 캡처 우선 + 서버 폴백 시 residential proxy. Data API를 쓸 경우 30일 저장 조항 때문에, 비공식 경로(oEmbed+확장 캡처) 중심 설계가 오히려 약관 충돌이 적다.
- 원본 영상 보존은 셀프호스트 파워유저용 옵션(사용자 자신의 yt-dlp 연동) 정도로만 로드맵에 남겨둘 것.

## 4. 일반 웹페이지

이 영역만은 문제가 없다. 성숙한 오픈소스 스택이 그대로 쓸 수 있는 상태.

### 본문 추출 (reader mode 계열)
- **Defuddle** (kepano/defuddle, Obsidian 제작자 개발): Mozilla Readability의 현대적 대체재. Readability가 사실상 방치 상태에서 약한 부분(MathJax/KaTeX 수식, 코드블록 하이라이팅, 각주, SPA 페이지)을 처리하고 마크다운 변환 품질이 좋다. Obsidian Web Clipper의 실제 파이프라인이 Defuddle + Turndown(GFM). 2025년 공개, 활발히 유지.
- Readability는 뉴스형 기사에는 여전히 준수하나 Reddit/X 등 SPA형 페이지에서 품질 저하 보고.

### 완전 보존 (원본 fidelity 계열)
- **SingleFile** (gildas-lormeau): CSS·이미지·폰트까지 인라인한 단일 HTML 저장. 브라우저 확장 + CLI 양쪽 제공, 2026년 2월에도 릴리스가 나온 활발한 프로젝트(AGPL-3.0 — 상용 통합 시 라이선스 검토 필요).
- **monolith** (Rust CLI): 서버 사이드 단일 HTML 번들링, 안정 상태.
- **ArchiveBox**: SingleFile+wget+readability+yt-dlp를 묶은 셀프호스트 아카이빙 스위트(MIT) — 파이프라인 설계의 좋은 참고 사례.

### 권장 접근
- **2계층 저장**: (a) Defuddle로 추출한 마크다운 본문 — AI 분류·검색·읽기 뷰용, (b) SingleFile(확장에서) 또는 monolith(서버에서)로 만든 단일 HTML 스냅샷 — 원본 fidelity 보존용. 페이월/로그인 페이지는 확장 경로에서만 캡처 가능하므로, 여기서도 확장이 1급 시민.

## 5. 음악 (Spotify / Apple Music 링크)

### Spotify
- **경고**: 2024년 11월 27일 Spotify가 신규 앱의 audio-features, audio-analysis, recommendations, related-artists, 30초 preview URL 접근을 하루아침에 차단(기존 승인 앱만 유지, 신규 경로 없음). 즉 "무드 기반 분류" 같은 오디오 특성 활용은 신규 앱에선 불가.
- 그러나 **기본 메타데이터는 문제 없음**: Web API의 track/album/artist 조회는 여전히 가능하고, 더 가볍게는 **oEmbed(`open.spotify.com/oembed`)가 무인증**으로 제목·썸네일·embed HTML을 반환.

### Apple Music
- Apple Music API + MusicKit: 카탈로그 검색·곡/앨범/아티스트 메타데이터 제공. 개발자 토큰(최대 6개월 유효 JWT) 필요 — Apple Developer Program 등록 전제(연 $99로 알려져 있으나 이번 검색에서 금액 원문은 미확인). 카탈로그 조회 자체는 사용자 구독 불필요.

### 크로스플랫폼 매칭
- **Odesli(Songlink) API**: 무료, 키 없이 분당 10회(키는 developers@song.link로 요청). 아무 플랫폼 링크를 넣으면 ISRC 기반 매칭으로 전 플랫폼 URL + 제목/아티스트/아트워크 반환. "Spotify 링크를 저장했는데 사용자는 Apple Music 유저" 문제를 한 방에 해결.

### 권장 접근
- **저장 시: Spotify/Apple oEmbed 또는 각 공식 API로 메타데이터+아트워크 확보 → Odesli로 크로스플랫폼 URL 세트 보강 → 아트워크 이미지는 자체 스토리지 복제.** 음원 파일 저장은 명백한 저작권 침해라 논외. 오디오 특성 기반 분류는 Spotify 경로가 막혔으므로 메타데이터(장르·아티스트)+AI 텍스트 분류로 대체.

## 미확인/추가 검증 필요 사항
1. X pay-per-use 단가의 공식 원문(docs.x.com) 대조 — 3rd-party 소스 다수 일치로 신뢰도는 높으나 공식 확인 전.
2. Apple Developer Program 연 $99 — 통념이나 이번 검색 결과에는 금액이 안 나옴.
3. LinkedIn ToS의 "사용자 본인용 브라우저 확장" 허용 범위 — 법률 자문 필요 영역.
4. 국내(한국) 법 기준의 스크래핑/사적복제 판례는 이번 조사 범위에서 다루지 않음 — 미국 판례 중심.
5. iOS 공유시트 경유 캡처의 실제 데이터 확보 범위(URL 외에 본문 텍스트가 넘어오는지)는 플랫폼별 실측 필요.

## Key Takeaways
- 아키텍처 대전제: X·LinkedIn·YouTube 모두 서버 사이드 수집이 차단·소송·약관으로 막혀 있으므로, 캡처는 사용자의 브라우저 확장(+모바일 공유시트)에서 수행하고 서버는 저장·미디어 복제·AI 분류만 담당하는 구조로 설계해야 한다.
- X 공식 API는 2026년 2월부터 free tier 폐지 + pay-per-use(읽기 $0.005/건) 전환 — 개인 아카이브 서비스의 기본 경로로는 부적합하고, 현실 경로는 확장 기반 DOM 캡처(1순위) + oEmbed/FxTwitter API 폴백(2순위)이다.
- X 미디어(이미지·영상)는 캡처 시점에 즉시 자체 스토리지로 복제해야 한다 — syndication/FxTwitter API가 미디어 직링크를 주지만 비공식이라 언제든 끊길 수 있다.
- LinkedIn은 서버 사이드 스크래핑 절대 금지 영역: hiQ는 ToS 위반으로 최종 패소($500K+영구금지), Proxycurl은 2025년 소송으로 폐업 — 사용자 본인 세션의 확장 캡처가 유일한 현실 경로이며 이마저 ToS 해석에 법률 자문이 필요하다.
- YouTube는 영상 원본 저장을 기본 기능에서 제외하고 메타데이터+썸네일+자막만 저장하는 것이 정답 — yt-dlp는 PO Token·데이터센터 IP 차단으로 서버 운용이 불안정하고, Yout v. RIAA로 DMCA 리스크가 확인됐으며, 공식 API는 30일 초과 저장을 금지해 아카이브 컨셉과 충돌한다.
- 유튜브 자막도 클라우드 IP에서 차단되므로(youtube-transcript-api RequestBlocked) 확장에서 사용자 세션으로 추출하는 것이 안정적이고, 서버 폴백은 residential proxy 비용을 전제해야 한다.
- 일반 웹페이지는 리스크 없음: Defuddle(마크다운 본문, AI 분류용) + SingleFile/monolith(단일 HTML 스냅샷, 원본 보존용)의 2계층 저장이 바로 구현 가능한 성숙 스택이다. 단 SingleFile은 AGPL-3.0이라 통합 방식 검토 필요.
- 음악 링크는 가장 쉬운 영역: Spotify oEmbed(무인증)+Apple Music API로 메타데이터, Odesli API(무료, 10req/min)로 크로스플랫폼 링크 매칭. 단 Spotify audio-features는 2024년 11월부터 신규 앱 차단이라 무드 분석 기능은 설계에서 빼야 한다.
- 프리미엄/원가 설계 시사점: X API·residential proxy·스토리지 등 원가가 발생하는 경로는 전부 폴백/프리미엄으로 밀고, 무료 사용자 기본 흐름은 원가 0인 확장 캡처+oEmbed로 구성 가능하다.
- 차별화 포인트 검증됨: 기존 툴 중 API 기반(Readwise)은 X만, 확장 기반(dewey/Twillot)은 북마크 내보내기에 그침 — '확장 캡처 + 콘텐츠 영구 보존 + AI 분류'를 묶은 통합 제품 포지션은 실제로 비어 있다.
