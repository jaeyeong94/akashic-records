# AI 자동 분류·검색 파이프라인 설계 사례 조사 (2026-07 기준)

## 1. 오픈소스 AI 태깅 파이프라인 실제 구현 사례

### Karakeep (구 Hoarder) — 가장 참고할 만한 레퍼런스
아카식 레코드와 컨셉이 거의 동일한 오픈소스(링크·노트·이미지 북마킹 + AI 자동 태깅 + 전문 검색). 구조:

- **스택**: Next.js 프론트 + 백그라운드 워커(비동기 잡 큐, `OpenAIQueue`) + Meilisearch(검색 인덱스) + LLM 프로바이더 추상화 레이어(`InferenceClient`)
- **AI 처리 흐름**: 링크 저장 → 크롤러가 본문 추출 → 잡 큐에 태깅/요약 작업 등록 → LLM 호출 → 태그·요약 저장 → Meilisearch 재색인. **AI 처리는 전부 비동기**로 저장 UX를 블로킹하지 않음
- **모델 구성** (환경변수로 교체 가능):
  - 텍스트 태깅: `INFERENCE_TEXT_MODEL` (기본 gpt-4o-mini급 소형 모델)
  - 이미지 태깅: `INFERENCE_IMAGE_MODEL` (기본 `gpt-4o-mini`, **저해상도 모드로 이미지 크기 무관 고정 토큰** 사용해 비용 통제)
  - 임베딩: `EMBEDDING_TEXT_MODEL` (기본 `text-embedding-3-small`)
  - OpenAI 호환 API(Gemini, OpenRouter, Azure)와 Ollama(로컬) 모두 지원
- **프롬프트 전략**: `constructTextTaggingPrompt`가 "태그 다양성 확보, 콘텐츠 언어 따르기, 보일러플레이트/에러 콘텐츠 제외" 규칙을 강제. 태그 표기 스타일(lowercase-hyphens / camelCase / titlecase-spaces)을 사용자 설정으로 정규화. 사용자가 커스텀 지시문을 추가할 수 있고 `$tags`, `$aiTags`, `$userTags` 플레이스홀더로 기존 태그 목록을 프롬프트에 주입(기존 태그 재사용 유도 → 태그 파편화 방지)
- **출력 형식**: `INFERENCE_OUTPUT_SCHEMA`로 structured(Zod 스키마)/json/plain 선택. 구조화 출력이 기본
- **컨텍스트 제한**: `INFERENCE_CONTEXT_LENGTH` 기본 2,048토큰으로 본문을 잘라서 투입 — 품질과 비용의 트레이드오프를 명시적 노브로 노출
- **요약**: 3~4문장, 사용자 선호 언어로 생성(`constructSummaryPrompt`)
- **OCR**: 기본 Tesseract, `OCR_USE_LLM=true`면 비전 모델로 구조 보존 OCR
- **공식 비용 문서**: 텍스트 태깅은 gpt-4o-mini급으로 **북마크 3,000개+/1달러**, 이미지 태깅은 저해상도 모드로 **이미지 1,000개+/1달러**

### Linkwarden
북마크 + 풀페이지 아카이빙 + AI 태깅. Ollama 로컬 모델(phi3:mini 같은 초경량 모델로 충분하다고 문서화) 또는 OpenAI/Anthropic 외부 API 선택. 특징: **"기존 태그에 자동 배정" 모드** 지원 — 자유 태깅과 고정 분류의 절충. 2.10부터 기존 링크 일괄 AI 태깅 지원.

### 공통 패턴 요약
1. 저장 즉시 응답, AI는 잡 큐로 비동기 처리
2. 소형·저가 모델(gpt-4o-mini급)로 충분 — 태깅은 프론티어 모델 불필요
3. 프로바이더 추상화(OpenAI 호환 + Ollama)로 셀프호스트 사용자 배려
4. 본문 트렁케이션으로 비용 상한 고정
5. 구조화 출력(JSON/Zod)으로 파싱 안정성 확보

### pgvector vs 전용 벡터DB
2026년 컨센서스: **개인 아카이브 규모(수만~수백만 벡터)에서는 pgvector가 정답**.
- pgvector: 5,000만~1억 벡터까지 프로덕션 검증. 별도 서비스 불필요, 트랜잭션 일관성, 메타데이터 필터링을 SQL로. HNSW 인덱스로 99% recall에서 sub-100ms
- Qdrant: Rust+SIMD로 p95 레이턴시 ~39% 우위, 샤딩·복제 수평확장. 그러나 이 이점은 수천만 벡터+고QPS에서만 의미 있음
- 개인 아카이브는 사용자당 수천~수만 항목이므로 pgvector(특히 Supabase 사용 시)로 시작하고, 병목이 실측되면 이전하는 것이 표준 조언

### 하이브리드 검색 (BM25/FTS + 벡터)
- 벡터 단독 검색은 고유명사·정확 키워드에 약함. 벤치마크: WANDS 이커머스에서 하이브리드가 단독 대비 NDCG +7.4%. 한 사례에서는 순수 벡터 62% → 하이브리드+RRF 84% 검색 정밀도
- **RRF(Reciprocal Rank Fusion)가 사실상 표준 융합 방식** — 점수 정규화 문제를 회피하고 순위만 사용. `k` 상수(Supabase 기본 50)로 상위권 가중 조절
- Supabase 공식 하이브리드 검색 가이드 존재: `tsvector` 생성 컬럼(GIN 인덱스) + `vector` 컬럼(HNSW 인덱스), 두 검색을 병렬 실행 후 SQL 함수에서 RRF 병합
- 프로덕션 스택의 정석: BM25 + 벡터 → RRF → (선택) 크로스인코더 리랭킹. 개인 아카이브 규모에선 리랭킹 생략해도 무방
- 대안: Karakeep이 쓰는 Meilisearch는 임베더만 설정하면 문서 임베딩 자동 생성 + 키워드/시맨틱 병렬 실행 + 자동 병합까지 내장(하이브리드를 기본 경험으로 만드는 로드맵 진행 중) — 검색 인프라를 직접 만들지 않는 선택지

## 3. 자동 카테고리 체계: 3가지 접근

### (a) 고정 분류체계 (fixed taxonomy)
- LLM에 분류체계를 주고 "이 중에서 골라라" 방식. 일관된 브라우징·네비게이션 제공, 태그 파편화 없음
- 단점: 분류체계 설계·유지 비용, 새 콘텐츠 유형에 경직
- 실사례: Linkwarden의 "기존 태그에 배정" 모드, 엔터프라이즈 자동분류(taxonomy를 LLM 가이드 입력으로 사용)

### (b) LLM 자유 태깅 (open tagging)
- 실사례: Karakeep, mymind. 콘텐츠마다 LLM이 자유롭게 태그 생성
- 장점: 초기 투자 zero, 콘텐츠 적응력 최고
- 단점: **단일 콘텐츠에만 적용되는 초구체적 태그가 양산**되어 태그 파편화 → 브라우징 축으로는 부적합. Karakeep이 기존 태그 목록을 프롬프트에 주입하는 것이 이 문제의 완화책
- mymind의 접근: 폴더·수동 태그를 아예 없애고 "저장하면 AI가 콘텐츠 유형(아티클/제품/책/레시피)·색상·브랜드·객체를 자동 인식, 찾기는 전부 검색으로" — 분류를 브라우징 UI가 아니라 **검색 리콜 향상 수단**으로만 사용

### (c) 클러스터링 기반 자동 카테고리
- 임베딩 클러스터링(HDBSCAN/BERTopic) 후 LLM이 클러스터 이름 부여
- 장점: 라벨 데이터 불필요, 사용자 데이터에서 카테고리가 창발
- 단점: 클러스터 품질 불안정, 통계적 동시출현 의존으로 오해적 그룹 생성 가능, 새 항목 추가 시 카테고리 재편 문제
- **Microsoft TnT-LLM (KDD 2024, Bing Copilot 로그에 적용)**: 2단계 프레임워크 — ①LLM이 샘플에서 분류체계 자체를 생성 ②그 분류체계로 경량 분류기 학습해 대규모 분류. 임베딩 클러스터링+클러스터 라벨링 베이스라인보다 정확도·해석가능성 모두 우위로 보고됨. "LLM으로 분류체계를 만들고, 만든 체계로 분류한다"는 하이브리드가 학계·업계 방향

### 실무 권고 종합
초기: LLM 자유 태깅 + 기존 태그 주입(Karakeep 방식) + 콘텐츠 유형(아티클/이미지/영상/음악 등)만 고정 축으로. 데이터가 쌓이면 TnT-LLM식으로 사용자별 분류체계를 주기적으로 재생성하는 2단계 진화가 현실적.

## 4. 이미지·영상·음악 콘텐츠 AI 이해 실전 패턴

### 이미지
- **비전 LLM 하나로 설명+태깅+OCR 통합**이 2026 표준. Karakeep도 `buildImagePrompt`(주제 분석)와 `buildOCRPrompt`(구조 보존 텍스트 추출)를 같은 비전 모델로 처리
- 비용: 저해상도 모드 고정 토큰 사용 시 이미지당 $0.001 수준. Gemini Flash 계열은 "6,000페이지/1달러" 수준의 OCR 비용 보고
- 정밀 OCR이 핵심이면 특화 모델(GLM-OCR, PaddleOCR-VL)이 프론티어 LLM보다 벤치마크 우위라는 2026 평가 있음 — 개인 아카이브 용도로는 범용 비전 LLM으로 충분
- 이미지 검색: 비전 LLM이 생성한 설명 텍스트를 텍스트 임베딩(단순, 저비용) vs 멀티모달 임베딩으로 이미지 직접 임베딩(색·분위기 검색 가능). mymind는 객체·색상·브랜드 태그 추출 방식

### 유튜브/영상
표준 파이프라인(다수 오픈소스 확인):
1. **1차: 기존 자막 활용** — youtube-transcript-api 등으로 업로더/자동생성 자막 추출(무료, 즉시)
2. **폴백: yt-dlp로 오디오 추출 → Whisper 계열 STT** (자막 없는 경우만)
3. 긴 영상은 청크 분할 → Map-Reduce 요약(청크별 요약 후 병합)
4. 타임스탬프 포함 키포인트 추출이 일반적 출력 형식

### 음악/오디오
- **CLAP 계열 임베딩**(오픈, 512차원)이 텍스트↔오디오 공유 벡터공간 표준. "차분한 피아노곡" 같은 자연어로 음원 검색 가능, 제로샷 분류 지원. MuQ-MuLan 등 후속 모델 존재
- 메타데이터 추출: 장르·악기·무드·템포·키를 ML로 추출하는 것이 음원 산업 표준 패턴
- 팟캐스트 등 발화 오디오는 STT 후 텍스트 파이프라인 재사용: gpt-4o-mini-transcribe $0.003/분, Deepgram Nova-3 배치 $0.0036/분, Groq Whisper-large-v3-turbo ~$0.0006/분

## 5. 항목 1개 저장 시 AI 처리 비용 추정

전제: 소형 모델(gpt-4o-mini/gpt-5-mini/Gemini Flash급, 입력 $0.15~0.30 / 출력 $0.60~2.50 per 1M토큰), 본문 2,000토큰 트렁케이션, 임베딩 text-embedding-3-small($0.02/1M).

| 콘텐츠 유형 | 처리 내용 | 추정 비용/건 |
|---|---|---|
| 웹 링크/게시물 | 태깅+요약(입력 2K+출력 300토큰) + 임베딩 | **$0.001~0.002** |
| 이미지 | 비전 태깅+설명(저해상도 고정 토큰) + OCR + 임베딩 | **$0.001~0.003** |
| 유튜브(자막 있음) | 자막 추출(무료) + 요약(20분 영상 자막 ~5K토큰) + 임베딩 | **$0.002~0.005** |
| 유튜브(자막 없음, 20분) | STT $0.003/분×20 + 요약 | **$0.06~0.07** (STT가 지배적) |
| 음악(메타데이터+CLAP) | CLAP 임베딩(셀프호스트 GPU 컴퓨트) | API 비용 ≈ 0, 인프라 비용만 |

- 실측 근거: Karakeep 공식 문서 "텍스트 3,000건+/1달러, 이미지 1,000건+/1달러" → 건당 $0.0003~0.001
- **결론: 텍스트·이미지·자막영상은 1,000건당 $1~5 수준으로 사실상 무시 가능. 비용 리스크는 자막 없는 긴 영상/오디오의 STT뿐** → "자막 우선, STT는 옵트인" 설계가 정석
- 추가 절감: 배치 API(OpenAI/Anthropic 50% 할인), 프롬프트 캐싱(반복 시스템 프롬프트 최대 90% 절감)

## 확인하지 못한 부분 / 주의사항
- 2026년 최신 모델명·가격(GPT-5.5, Gemini 3.5 Flash, Claude Opus 4.8 등)은 서드파티 가격 비교 사이트 기준이며 각 사 공식 가격 페이지로 교차 검증하지 못함. 실제 구현 시 공식 문서 재확인 필요
- Karakeep의 기본 텍스트 모델명이 소스(DeepWiki)에 "gpt-4-mini"로 표기되어 있으나 이는 gpt-4o-mini/gpt-4.1-mini의 오기로 보임 — 저장소 코드로 직접 확인하지 못함
- Gemini 임베딩의 이미지·영상·오디오 임베딩 GA 여부는 공식 문서로 확인하지 못함(서드파티 기사만 존재)
- mymind는 클로즈드소스라 내부 파이프라인(모델·프롬프트)은 공개 정보 없음 — UX 패턴만 참고 가능
- 벤치마크 수치(NDCG +7.4%, 62%→84% 등)는 개별 블로그/벤더 사례이며 도메인마다 달라질 수 있음

## Key Takeaways
- 아키텍처 정석은 Karakeep 패턴: 저장 즉시 응답 + AI 처리(태깅·요약·임베딩)는 잡 큐 비동기, 본문은 2K토큰 수준으로 잘라 비용 상한 고정, 구조화 출력(JSON 스키마)으로 파싱 안정화
- 태깅·요약에 프론티어 모델 불필요 — gpt-4o-mini/Gemini Flash급 소형 모델로 텍스트 3,000건/1달러, 이미지 1,000건/1달러 수준 (Karakeep 공식 실측)
- 태그 파편화가 LLM 자유 태깅의 최대 함정 — 기존 태그 목록을 프롬프트에 주입해 재사용을 유도(Karakeep 방식)하고, 콘텐츠 유형(아티클/이미지/영상/음악)만 고정 축으로 두는 하이브리드가 현실적
- 카테고리 자동 생성의 장기 방향은 TnT-LLM 패턴(Microsoft, Bing Copilot 적용): LLM이 사용자 데이터에서 분류체계를 생성하고 그 체계로 대량 분류 — 순수 클러스터링보다 해석가능성·정확도 우위
- 검색은 하이브리드(BM25/FTS + 벡터 + RRF)가 필수 — 벡터 단독은 고유명사·정확 키워드에 약하며, Supabase가 pgvector+tsvector+RRF 공식 가이드를 제공하므로 구현 장벽 낮음
- 벡터DB는 pgvector로 시작이 정답 — 개인 아카이브 규모(사용자당 수천~수만 항목)에서 전용 벡터DB(Qdrant 등)의 성능 이점은 무의미하고 운영 복잡도만 증가; 5,000만 벡터까지 프로덕션 검증됨
- 임베딩 모델: 가성비는 text-embedding-3-small($0.02/1M), 한국어 포함 다국어 품질은 gemini-embedding-001($0.15/1M, Matryoshka로 차원 절감 가능)이 우위 — 다국어 사용자 대상이면 후자 검토
- 유튜브는 '자막 우선, STT 폴백' 설계가 비용의 핵심 — 자막 있으면 건당 $0.005 이하, 자막 없는 20분 영상 STT는 건당 $0.06+로 10배 이상 차이. STT는 옵트인으로
- 음악 검색은 CLAP 계열 오픈 임베딩(텍스트↔오디오 공유 벡터공간)으로 자연어 음원 검색 가능 — API 비용 없이 셀프호스트 가능하나 GPU 인프라 필요, MVP에서는 메타데이터 태깅만으로 시작 가능
- 항목당 AI 처리 비용은 텍스트·이미지 기준 $0.001~0.003으로 사실상 무시 가능 — 비용이 아니라 태그 품질(파편화 방지)과 파이프라인 안정성(비동기·재시도)이 진짜 설계 과제
