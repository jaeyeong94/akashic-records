# 아카식 레코드 (Akashic Records) — 기획서

> 개인의 우주 도서관. 기록할 가치가 있는 모든 것을 저장하면, 링크가 죽어도 내용은 남고, AI가 사서가 되어 정리·검색·재발견을 대신한다.

- 작성일: 2026-07-13
- 상태: v0.3 (기획 대화 2차 반영 — 명칭·확장 범위·잡 큐·다음 단계 확정)
- 근거 자료: [경쟁 제품 분석](research/01-competitors.md) · [아카이빙 기술 현실성](research/02-archiving-feasibility.md) · [AI 파이프라인 사례](research/03-ai-pipeline.md)

---

## 1. 문제 정의

1. **저장한 것이 사라진다.** 링크는 썩고(link rot), 트윗은 삭제되고, 계정은 폐쇄되고, 서비스는 죽는다. Pocket은 2025년 7월 종료 후 3개월간 CSV 내보내기만 허용한 뒤 2천만 사용자의 누적 20억 건 저장물을 전량 삭제했다.
2. **저장한 것을 못 찾는다.** 북마크·좋아요·스크랩이 플랫폼별로 흩어져 있고(X 북마크, LinkedIn 저장, 유튜브 나중에 보기, 브라우저 북마크…), 폴더 정리는 노동이라 결국 안 하게 되며, 키워드 검색은 "그때 본 그거"를 못 찾는다.
3. **소셜 콘텐츠는 보존 자체가 어렵다.** X·LinkedIn은 로그인 장벽 + API 유료화/폐쇄로, 기존 범용 아카이브 툴이 원문을 온전히 보존하지 못한다.

## 2. 제품 비전

**"모든 개인 기록물의 단일 아카이브 + AI 사서"**

- 사용자가 LinkedIn·X 게시물, 웹 링크, 이미지, 미디어 파일, 유튜브, 음악 등을 한 곳에 저장한다.
- 콘텐츠 **자체**를 보존한다 — 원본 링크가 죽어도 내 도서관에는 남는다.
- AI가 자동으로 분류·체계화하고(자동 정리), 자연어로 찾아주고(검색), 관련 항목을 이어주고(재발견·연결), 요약해준다(소화).

### 대상 사용자
- **1단계: 개인용(제작자 본인).** 온보딩·과금·멀티유저 UX 없이 기능에 집중. 단, 배포형 웹서비스이므로 **단일 계정 인증(Supabase Auth)은 MVP 첫날부터 유지** — 인증 게이트 없는 배포는 아카이브 전체 공개 열람 + 인제스트 엔드포인트의 익명 쓰기 구멍이 된다.
- **2단계: 제품화 여지.** 멀티유저 확장이 가능한 구조로 설계(멀티테넌시를 막는 결정 금지). 출시 여부는 1단계 사용 경험 후 판단.

## 3. 시장 포지션과 차별점

리서치로 확인된 빈 포지션 (상세: [경쟁 분석](research/01-competitors.md)):

| 차별점 | 근거 |
|---|---|
| **소셜 포스트 원문 보존 + 범용 아카이브의 결합** | dewey는 소셜 전용, Raindrop·mymind·Readwise는 로그인 벽 뒤 소셜 콘텐츠를 온전히 보존 못 함. 결합한 제품은 없음 |
| **콘텐츠 보존이 기본값** | 기존 툴은 보존을 최상위 유료 플랜에 가두거나(mymind Mastermind, Raindrop Pro) 셀프호스팅 요구(Karakeep). Pocket 사망 이후 "내 저장물은 내 것" 불안이 시장 전반에 존재 |
| **완전한 export 보장** | mymind의 export 부재, dewey의 유료 Export Pass가 대표적 불만. 신뢰 서사로 활용 |
| **한국어 1급 지원** | 전 제품의 공백. 한국어 콘텐츠의 AI 분류·요약·검색 품질 보장 |
| **음악·미디어 파일을 1급 객체로** | 음악 링크를 제대로 다루는 아카이브 툴 없음 |

AI 자동 태깅·시맨틱 검색 자체는 이미 업계 표준(table stakes)이다. 차별화는 그 위층 — 보존의 신뢰성, 소셜 커버리지, 연결·재발견의 품질에서 만든다.

## 4. 아키텍처 대전제 (리서치로 확정)

X·LinkedIn·YouTube 모두 서버 사이드 수집이 붕괴했다 (상세: [아카이빙 현실성](research/02-archiving-feasibility.md)):

- X: 무료 API 폐지(2026.2), pay-per-use 전환. Nitter 생태계 사망.
- LinkedIn: 스크래핑 업체를 소송으로 폐업시키는 플랫폼(hiQ 최종 패소, Proxycurl 2025 폐업). 서버 사이드 수집은 상용 서비스로서 감수 불가능한 리스크.
- YouTube: PO Token 강제 + 데이터센터 IP 광범위 차단. 영상 다운로드 서비스는 DMCA 표적(Yout v. RIAA).

따라서:

> **로그인 장벽 콘텐츠의 완전 캡처는 사용자의 브라우저(확장/공유시트)에서 수행하고, 서버는 저장·미디어 복제·AI 분류만 담당한다.**
> 서버 사이드 인제스트가 가능한 범위는 일반 웹페이지, 음악 메타데이터, 그리고 X·유튜브의 무인증 메타데이터(oEmbed)까지다.

법적으로도 "사용자 본인이 보는 콘텐츠를 본인이 저장"하는 구조라 서버 스크래핑과 리스크 등급이 다르다.

## 5. 저장 정책 — 항목별 차등

| 소스 | 저장 내용 | 확보 경로 |
|---|---|---|
| X 게시물 | 텍스트·작성자·메타데이터 + 이미지/영상 원본 즉시 자체 스토리지 복제 | 1순위 확장 DOM 캡처, 폴백 oEmbed → FxTwitter API |
| LinkedIn 게시물 | 텍스트·작성자·이미지·메타데이터 | 확장 캡처 단일 경로 (서버 수집 전면 배제) |
| 일반 웹페이지 | 2계층: ① Defuddle 마크다운 본문(AI·검색·읽기용) ② 단일 HTML 스냅샷(원본 보존용) | 확장(자체 경량 캡처, 페이월 페이지 포함 — SingleFile은 AGPL 전파로 내장 불가, §8) 또는 서버(monolith) |
| 유튜브 | 메타데이터·썸네일·자막/트랜스크립트. **영상 원본은 저장하지 않음**(DMCA·비용·불안정 삼중고) | oEmbed(메타) + 확장 자막 캡처, 서버 폴백은 residential proxy 전제 |
| 음악 (Spotify/Apple Music) | 메타데이터·아트워크(자체 복제)·크로스플랫폼 링크 세트 | oEmbed/공식 API + Odesli API 매칭 |
| 이미지·파일 업로드 | 원본 그대로 | 웹앱 드롭/공유시트 |

원칙: **미디어는 캡처 시점에 즉시 자체 스토리지로 복제한다** (원본 삭제·비공식 API 단절 대비).

'차등'의 의미: 사용자가 항목마다 보존 수준을 고르는 것이 아니라 **소스 유형별 자동 정책**이다.

## 6. 수집 경로 — 단계별

| 단계 | 경로 | 비고 |
|---|---|---|
| **Phase 1 (MVP)** | 웹앱 직접 붙여넣기·드롭 + **브라우저 확장** | 확장이 차별점(X·LinkedIn 보존)의 유일한 실현 경로이므로 처음부터 포함 |
| Phase 2 | 텔레그램 봇 | iOS PWA 공유시트 제약 때문에 모바일 캡처의 현실적 1착. URL만 받은 소셜 포스트는 "다음에 데스크톱에서 열면 본문 캡처" 지연 캡처 UX |
| Phase 3 | 모바일 공유 시트 (PWA/앱) | Android는 PWA share_target, iOS는 앱 필요 시 판단 |

저장 마찰 목표: **어떤 경로든 2초 이내 저장 완료 체감** (저장 즉시 응답, AI 처리는 비동기).

## 7. AI 파이프라인

(상세: [AI 파이프라인 사례](research/03-ai-pipeline.md), Karakeep 패턴 준용)

### 처리 흐름
저장 → 즉시 응답 → 잡 큐 비동기: 본문 추출 → 태깅·요약(소형 모델, 본문 ~2K 토큰 트렁케이션) → 임베딩 → 검색 인덱스 갱신. 구조화 출력(JSON 스키마) + 재시도.

저장 직후 항목은 색인 완료 전까지 검색에 잡히지 않으므로(최종 일관성), UI에 '처리 중' 상태를 노출한다.

### 4가지 가치의 구현
1. **자동 정리**: LLM 자유 태깅 + **기존 태그 목록을 프롬프트에 주입해 재사용 유도**(태그 파편화 방지 — 자유 태깅의 최대 함정). 콘텐츠 유형(글/게시물/이미지/영상/음악/파일)만 고정 축. 장기적으로 TnT-LLM 패턴(사용자 데이터에서 분류체계를 주기 재생성)으로 진화.
2. **자연어 검색**: 하이브리드 검색 — pgvector(HNSW) + Postgres 전문검색(tsvector) + RRF 융합. Supabase 공식 가이드 스택 그대로. 벡터 단독은 고유명사·정확 키워드에 약하므로 하이브리드 필수.
3. **재발견·연결**: 임베딩 유사도 기반 관련 항목 표시, 잊힌 저장물 리서페이싱. Phase 3 (데이터가 쌓여야 가치가 나옴).
4. **요약·소화**: 항목별 3~4문장 요약(사용자 언어로). 유튜브는 자막 기반 요약 — **자막 우선, STT는 옵트인**(자막 없는 20분 영상 STT는 건당 $0.06+로 10배 이상 비쌈).

### 모델·비용
- 태깅·요약: 소형 모델(Gemini Flash/gpt-4o-mini급)로 충분 — 프론티어 불필요. 텍스트 3,000건/1달러, 이미지 1,000건/1달러 (Karakeep 실측).
- 임베딩: 한국어 포함 다국어 품질 기준 gemini-embedding-001 우선 검토(Matryoshka 차원 절감), 가성비 대안 text-embedding-3-small.
- 이미지: 비전 LLM 하나로 설명+태깅+OCR 통합(저해상도 고정 토큰).
- **항목당 $0.001~0.003 — 비용이 아니라 태그 품질과 파이프라인 안정성이 진짜 과제.**

## 8. 기술 방향

- **웹앱**: Next.js (Vercel) — 개인용에서 멀티유저 SaaS까지 동일 스택.
- **데이터·스토리지·인증**: Supabase — Postgres(+pgvector, tsvector), Storage(미디어 복제), Auth(멀티테넌시 대비 RLS).
- **브라우저 확장**: Chrome MV3 단일 타깃 (WXT 프레임워크 — Firefox/Safari 확장 여지는 유지하되 MVP에서 지원하지 않음). DOM 캡처 + 페이지 스냅샷. 확장→서버 인증은 개인 액세스 토큰 또는 Supabase 세션.
- **아키텍처 제약**: SingleFile은 AGPL-3.0 — SaaS 제공만으로 카피레프트가 발동하므로 확장·서버에 라이브러리로 내장하지 않고, 별도 프로세스/CLI 격리 호출 또는 대체 구현으로만 통합한다.
- **비동기 처리**: Supabase Queues(pgmq) — 데이터와 동일 인프라, 잡 결과가 DB로 가는 구조라 운영 단순. 워커는 Edge Function/cron 트리거.
- 스키마는 처음부터 `user_id` 포함(멀티테넌시를 막는 결정 금지).

## 9. 로드맵

### Phase 1 — MVP: "저장하면 남는다" (웹앱 + 확장)

순서 원칙: 핵심 가치 루프(저장→보존→AI 정리→검색)의 검증이 확장 안정화에 인질 잡히지 않도록, 서버 사이드로 완결되는 1a를 먼저 완성하고 확장(1b)을 얹는다.

**Phase 1a — 웹앱 + 서버 인제스트** (전부 서버 사이드, 리스크 낮음)
- 단일 사용자 인증(Supabase Auth) — 배포형이므로 첫날부터
- URL 붙여넣기·파일 드롭 저장, 카드 그리드 라이브러리, 항목 상세(스냅샷 뷰)
- 일반 웹 2계층 저장, 유튜브 메타+자막, 음악 메타+Odesli, X 폴백 경로(oEmbed/FxTwitter)
- AI: 자동 태깅+요약+임베딩 비동기 파이프라인, 하이브리드 검색
- Export: 전체 데이터 내보내기, 재임포트 가능한 형식 (신뢰 서사의 기본)

**Phase 1b — 브라우저 확장** (차별점)
- X 게시물 원클릭 캡처(미디어 즉시 복제 포함) — 캡처 실패 시 oEmbed 폴백 자동 전환
- 일반 페이지 스냅샷(페이월 페이지 포함)
- 캡처 완결성 검증: 본문 비어있음 등 캡처 실패를 감지해 사용자에게 즉시 경고
- **LinkedIn 캡처는 X 캡처 안정화 이후** — 폴백 없는 단일 경로 + 법적 리스크 최상위 플랫폼을 두 번째로 진다

**Phase 1 수용 기준**
1. X·일반웹·유튜브·음악·이미지·파일 6종의 저장이 각각 성공하고, 원본 링크가 접근 불가여도 상세뷰에서 내용이 복원된다.
2. 자연어 질의(한국어 포함) 상위 결과에 목표 항목이 잡힌다.
3. 저장·캡처 실패가 조용히 삼켜지지 않고 사용자에게 감지된다.
4. 어떤 경로든 저장 조작 완료까지 체감 2초 이내 (AI 처리는 비동기).
5. 전체 export가 재임포트 가능한 형식으로 동작한다.

### Phase 2 — 모바일 캡처와 소화
- 텔레그램 봇 수집, 지연 캡처 UX
- 유튜브 STT 옵트인, 일일/주간 다이제스트(요약 소화)

### Phase 3 — 도서관 사서
- 관련 항목 연결, 리서페이싱, 사용자별 분류체계 자동 생성(TnT-LLM 패턴)
- 제품화 판단: 멀티유저 온보딩, 과금(무료 티어 + $5~10/월 벤치마크, 보존·스토리지를 유료 축으로 하되 핵심 가치를 최상위 플랜에 가두지 않기 — mymind 반면교사)

## 10. 하지 않을 것 (Out of Scope)

- 유튜브 영상 원본 저장 — 메타+썸네일+자막까지만 (셀프호스트 파워유저 옵션은 장기 로드맵으로만)
- 음원 파일 저장 — 명백한 저작권 침해
